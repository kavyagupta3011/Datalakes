# Generic Medallion Lakehouse

A **full-stack** generic Python ETL pipeline (Bronze → Silver
→ Gold → OLAP) that loads into PostgreSQL, a FastAPI backend, and a React dashboard for
uploading files, browsing the warehouse, and visualizing OLAP cubes with drill-through
back to source rows.

Nothing in the processing code is tied to any one business domain - the same engine
ingests retail, education, and support data side by side (and anything else dropped
into `bronze/`, including through the UI) using one config file for column aliases. No
per-dataset code anywhere in the pipeline.


## Tech stack

| Layer | Technology |
|---|---|
| Ingestion / ETL | Python 3.11, Pandas, PyArrow (Parquet) |
| Data warehouse | PostgreSQL, SQLAlchemy |
| OCR | Tesseract (via `pytesseract`), offline, no API key |
| Backend API | FastAPI, Uvicorn |
| Frontend | React 18, Vite 5, TypeScript 5, Tailwind CSS 3 |
| Charts / icons | Recharts, lucide-react |
| Testing | pytest, `pgserver` (embedded Postgres for CI) |
| Containerization | Docker, docker-compose, nginx (frontend reverse proxy) |
| CI | GitHub Actions |

## Project overview: how data flows through it

```
Bronze (raw files)  ->  Silver (cleaned, typed, partitioned Parquet)  ->  Gold (Postgres star schema)  ->  OLAP cuboids
        ^                                                                                                       |
        |___________________________________ React frontend  <->  FastAPI backend ____________________________|
                       (upload files, trigger pipeline runs, browse tables, drill into cubes)
```

1. **Bronze** — raw files land in `bronze/<domain>/<entity>_<date>.<ext>`, either dropped
   in by hand, generated by the synthetic data script, or uploaded through the UI.
2. **Silver** — every file is decoded, cleaned, type-inferred, and column names are
   harmonized to canonical names generically (no per-dataset mapping required). Output
   is partitioned Parquet plus lineage columns on every row.
3. **Gold** — Silver data is loaded into a real Postgres star schema: auto-classified
   dimension and fact tables, foreign keys resolved automatically, loaded
   incrementally by default.
4. **OLAP** — for every fact table, cuboids (grand totals, monthly rollups, per-dimension
   slices, and combinations) are materialized automatically, with a validator that
   independently recomputes the numbers to catch bugs.
5. **API + Frontend** — FastAPI serves all of this as JSON (plus upload and
   pipeline-trigger endpoints); the React app is where you actually look at it,
   upload new files, and click into a chart bar to see the exact source rows behind it.

## Features

**Bronze**
- Multi-format ingestion: CSV, Excel (multi-sheet, auto-skips empty sheets), JSON (flat + nested), XML (hierarchical)
- Lightweight file-based catalog (`catalog/catalog.json`): auto-discovery, duplicate detection, domain/entity inference from folder + filename convention
- Graceful fallback for files that don't follow convention (`domain="unknown"`, flagged for review instead of failing)

**Silver**
- Structural cleaning: column-name normalization, whitespace trimming, empty-row drop, exact-duplicate-row drop
- Type inference (boolean / numeric / datetime / string) using value-distribution thresholds, not per-column hardcoding
- Generic schema harmonization: alias config → heuristic fallback → persistent mapping memory (`mappings/column_mappings.json`) that gets smarter over time — resolved by *method*, not an arbitrary numeric confidence score
- Schema drift detection (added/removed columns vs. the previous run's profile)
- Per-file data-quality scorecard (`*.profile.json`): row/column counts, null %, completeness score, sample values
- Lineage columns on every row: `_bronze_path`, `_domain`, `_entity`, `_format`, `_file_checksum`, `_processed_at`, `_pipeline_run_id`, `_row_checksum`
- Partitioned Parquet output (`silver/domain=.../entity=.../year=.../month=...`), Snappy-compressed
- Multithreaded processing, idempotent re-runs, `processing_log.json` + `review_needed.json`
- Unstructured pass-through (images/audio/PDF copied as-is into the same partition scheme)
- Offline OCR on image/PDF attachments (Tesseract via `pytesseract`, no LLM, no API key): each file gets a `.ocr.json` sidecar with extracted text plus the same lineage columns; failures are caught and logged, never crash the run

**Gold (PostgreSQL)**
- Generic `dim_date` built from every date column found across all entities
- Auto-generated `dim_<entity>` tables for whatever's classified as dimension-like data (with a manual override option)
- Auto-generated `fact_<domain>_<entity>` tables for whatever's classified as transactional, with foreign keys resolved against the dimensions automatically
- Full lineage columns carried through into Gold — every fact/dim row traces back to its exact Bronze file and checksum
- Incremental loading by default (dimensions append only new natural-key rows, facts append only new rows by `_row_checksum`) — re-running the pipeline never duplicates data. `--full-reload` truncates and rebuilds instead

**OLAP**
- Generic cuboid materializer: for every fact table it builds an apex (grand total), a month rollup, a slice per dimension, and month × dimension drill combos — with no per-domain code
- `drill_to_source()` helper: go from a cuboid number back to the underlying fact rows (and from there, straight back to the Bronze file via lineage columns)
- `validate_olap.py`: recomputes every apex cuboid independently from the fact table and fails loudly on mismatch

**Orchestration**
- Dependency-free runner for the five pipeline steps, with per-step retry + exponential backoff, progress persisted to `catalog/orchestration_state.json`, and `--resume` to skip steps that already succeeded

**API**
- FastAPI backend over whatever `dim_/fact_/cuboid_` tables exist in Gold, plus a deliberately small write surface: upload a file into Bronze, and trigger/poll a pipeline run — no auth, this is a portfolio demo, not production

**Frontend**
- React + Vite + TypeScript + Tailwind dashboard (light purple/blue theme) — drag-and-drop upload into Bronze, one-click pipeline trigger with live step status, a table explorer for every dim/fact/cuboid table, and a cuboid viewer (apex totals, monthly trend chart, per-dimension bar charts) with click-to-drill back to the underlying fact rows and their lineage columns

## How an entity becomes a "dimension" vs. a "fact"

This is the one place genericness needs a pragmatic heuristic (documented, not hidden):

- **dimension**: looks like master data — one identifier column, no numeric measures, at most one descriptive date (e.g. customers, students, agents)
- **fact**: looks transactional — 2+ identifier columns (it references other entities), OR has numeric measures, OR has 2+ date columns suggesting an event with a duration (e.g. orders, grades, feedback, tickets)

It won't be perfect on every possible schema, but it works without any domain-specific code, which is the point.

**Overriding it**: if the heuristic gets a specific entity wrong, force it in `config/entity_overrides.yaml`:

```yaml
retail:
  customers: dimension
  orders: fact
```

Overrides are checked before the heuristic runs, per `domain.entity`, and logged when applied.

## Requirements

- **Python 3.11+**
- **PostgreSQL** (any reachable instance — local install, Docker, or remote)
- **Node.js 18+** and npm (only needed for the frontend)
- **Tesseract OCR** binary — optional, only needed if you want OCR text extraction on image/PDF uploads; the pipeline runs fine without it (OCR sidecars are written with an error field instead)
- **Docker + Docker Compose** — optional, only needed if you want the one-command setup instead of running each piece manually

## Setup and running — Option 1: Docker

```
docker compose up --build
```

This brings up four services in order: Postgres, a one-shot `pipeline` job (generates
synthetic data and runs the full pipeline, then exits), the FastAPI backend, and the
React frontend.

Open:
- `http://localhost:3000` — the dashboard
- `http://localhost:8000/docs` — raw API (Swagger UI)

Other useful commands:
```
docker compose run pipeline      # re-run just the pipeline (e.g. after uploading a file via the UI)
docker compose down              # stop everything
docker compose down -v           # stop everything AND wipe the Postgres/data volumes
```

The `api` and `pipeline` services share the same Bronze/Silver named volumes, so files
uploaded through the UI are visible to a pipeline run triggered through the UI.

## Setup and running — Option 2: Manual (no Docker)

**1. Install prerequisites:** Python, PostgreSQL, Node.js (see Requirements above).

**2. Create the database** (open "SQL Shell (psql)" on Windows, or `psql` elsewhere, and run):
```sql
CREATE DATABASE lakehouse;
```

**3. Install Python dependencies and configure the connection:**
```
pip install -r requirements.txt
cp .env.example .env
```
Fill in `.env`:
```
PG_HOST=localhost
PG_PORT=5432
PG_DB=lakehouse
PG_USER=postgres
PG_PASSWORD=<your password>
```

**4. Run the pipeline:**
```
python src/generate_bronze.py        # synthetic retail/education/support data, or drop your own files into bronze/
python src/build_catalog.py
python src/orchestrator.py
```
Useful orchestrator flags:
```
python src/orchestrator.py --resume                          # skip steps that already succeeded
python src/orchestrator.py --full-reload                     # rebuild Gold from scratch instead of incremental
python src/orchestrator.py --steps gold_olap validate_olap    # run a subset
python src/orchestrator.py --max-retries 3 --backoff-seconds 5
```
Or run each step individually (equivalent, no retry/resume):
```
python src/bronze_to_silver.py
python src/silver_to_gold.py     # add --full-reload to rebuild instead of incrementally append
python src/gold_olap.py
python src/validate_olap.py
```

**5. Start the API** (new terminal, leave running):
```
python src/api.py                 # http://localhost:8000, docs at /docs
```

**6. Start the frontend** (another new terminal, leave running):
```
cd frontend
npm install
cp .env.example .env              # set VITE_API_BASE_URL=http://localhost:8000
npm run dev                       # http://localhost:5173
```

**7. Verify it's working:**
```
curl http://localhost:8000/health
curl http://localhost:8000/tables
python src/validate_olap.py
```
All three should return data / exit cleanly with no errors.

## API reference

| Endpoint | Description |
|---|---|
| `GET /health` | DB connectivity check |
| `GET /tables` | list `dim_/fact_/cuboid_` tables, grouped |
| `GET /tables/{name}?limit=&offset=` | paginated rows from any table |
| `GET /drill/{fact_table}?<cuboid_keys>` | underlying fact rows for a cuboid slice, with lineage columns intact |
| `POST /upload` (form: `domain`, `entity`, `file`) | save a file into `bronze/<domain>/<entity>_<timestamp>.<ext>` |
| `POST /pipeline/run?full_reload=` | run the orchestrator in the background (409 if already running) |
| `GET /pipeline/status` | live + last-run step status, polled by the frontend |

No auth — this is a portfolio demo with a deliberately small write surface (upload +
pipeline trigger), not a production write API.

## Frontend views

- **Dashboard** — live table/cuboid counts, last pipeline run status
- **Upload** — drag-and-drop into Bronze + pipeline trigger with live step-status polling
- **Table Explorer** — browse every dim/fact/cuboid table, paginated
- **Cuboid Viewer** — apex stat cards, a monthly trend chart, per-dimension bar charts with dimension keys resolved to readable labels — click a chart point or bar to drill straight into the underlying fact rows and their lineage columns

Without `VITE_API_BASE_URL` set, the app calls a relative `/api` path, which only
resolves in Docker (where nginx proxies it to the `api` service); for local dev against
a bare `python src/api.py`, set the env var as shown above.

## Testing

```
pip install -r requirements-dev.txt
python -m pytest -v
```

- `tests/test_type_inference.py`, `tests/test_classification.py` — unit tests, no DB required
- `tests/test_incremental_load.py` — integration tests against a real Postgres (`PG_*` env vars / `.env`), marked `integration` and auto-skipped (not failed) when no database is reachable
- Run only unit tests: `python -m pytest -m "not integration"`
- 22 tests total; all pass against both a local Postgres and the embedded `pgserver` instance used in CI

## CI

`.github/workflows/ci.yml` runs on every push/PR to `main`: spins up a Postgres service
container, installs dependencies (including Tesseract for OCR), and runs the full
pytest suite (unit + integration).

## Onboarding a new domain

1. Drop files into `bronze/<your_domain>/<entity>_<YYYYMMDD>.<ext>`
2. (Optional) add an alias block to `config/domain_aliases.yaml` if you want specific
   column names recognized and auto-approved immediately — otherwise the heuristic
   fallback + mapping-memory review process handles it, you'll just see it show up in
   `mappings/column_mappings.json` as unapproved (resolved via a heuristic, not a
   curated alias)
3. Run the pipeline — new `dim_`/`fact_` tables and cuboids appear automatically, no
   code changes required

## Design choices / scope

- Orchestration is a single dependency-free script (subprocess + retry loop), not a DAG
  scheduler like Airflow — intentional, to keep the pipeline readable and free of
  heavyweight infra for a demo project
- Column harmonization trust is method-based (memory / alias / heuristic / fallback),
  not an arbitrary numeric confidence score — first matching rule wins, and only
  curated-alias or previously-approved mappings are auto-approved
- OCR is offline Tesseract only, no LLM-based extraction and no API key — keeps the
  pipeline runnable with zero external dependencies or cost
- The API's write surface is intentionally tiny (upload + pipeline trigger) and
  unauthenticated — enough to make the frontend genuinely interactive for a demo, not a
  general-purpose write API. The rest of the API stays read-only JSON; point a
  notebook or BI tool at it directly if you'd rather skip the UI
- Postgres instead of a lakehouse table format (Delta/Iceberg) — gives a real SQL star
  schema with FK semantics and ACID writes without needing a Spark cluster, appropriate
  for the scope of a single-engineer demo
